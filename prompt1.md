Project: Nakul

---

You are building a production-grade server intelligence and protection platform for a cPanel/WHM based hosting environment. The target stack includes CloudLinux OS, LiteSpeed Web Server, MySQL/MariaDB, Imunify360, Backuply Server Backups, CSF, Softaculous Installer, and WP Toolkit. The system must run as a persistent system-level agent that starts automatically when the server boots, operates 24/7, and continuously monitors the server state, service health, resource usage, security events, abuse patterns, backups, and database behavior. It must generate human-readable insights, actionable alerts, and a live dashboard accessible from the server web interface on port 8122.

The goal is to create a single-command deployable solution that can be installed on a fresh or existing server using one bootstrap script. That bootstrap script must install dependencies, create directories, write config files, generate the backend service, create the web dashboard, configure the systemd service, set permissions, and start everything automatically. The final system must be modular, resilient, and optional-service aware. If any monitored service is missing, disabled, not installed, or misconfigured, the agent must not crash. Instead, it must flag the missing dependency, mark that module as unavailable, and continue monitoring all other components. The configuration must allow the user to enable or disable each integration independently and change log file paths, scan intervals, alert thresholds, retention settings, and dashboard options without editing code.

Build the system as a modern monitoring and analysis platform with these major capabilities. It must read and correlate logs from multiple sources, including but not limited to cPanel/WHM logs, Apache/LiteSpeed logs, PHP error logs, CloudLinux-related signals, MySQL/MariaDB logs, CSF logs, Imunify360 logs, Backuply logs, Softaculous logs, WP Toolkit logs, system journal logs, kernel logs, auth logs, and relevant cPanel account/domain data. It should identify abusive behavior such as excessive CPU, memory, I/O, entry processes, process count, disk activity, connection floods, brute-force attempts, malware indicators, suspicious PHP execution patterns, database abuse, abnormal query load, mail abuse signals where available, and repeated resource-limit breaches. It should distinguish between noise and meaningful incidents. The output should be written for humans, not just machines, so each alert must include what happened, why it matters, what evidence was found, which account or domain may be involved, which logs were consulted, and what the recommended next action is.

The agent should maintain a live inventory of the server state. It must detect whether LiteSpeed, MySQL/MariaDB, Imunify360, CSF, Backuply, Softaculous, WP Toolkit, and CloudLinux are installed and running. For each service, the dashboard must show status, version if available, last check time, current health, recent events, and any recent alerts. For example, if a log path does not exist or a service is missing, the UI should show “Not Installed,” “Unavailable,” or “Path Not Found” instead of failing. If a service is present but inaccessible due to permissions, the system should record the error and recommend corrective action. All log-reading paths, binary paths, config locations, and service names must be configurable from a central config file. Use sane defaults for common installations, but never hardcode assumptions that cannot be overridden.

The system must include intelligent detection logic. It should be able to correlate high CPU usage with specific cPanel users, domains, PHP processes, cron jobs, web requests, or MySQL activity. It should correlate web attacks with request patterns, IP bursts, user agents where available, blocked events, and abnormal access log frequency. It should correlate database abuse with slow queries, high processlist activity, excessive connections, long-running queries, or repeated query patterns tied to a user or application. It should identify resource-limit breaches from CloudLinux LVE signals, top processes, and kernel or system performance indicators. It should detect DDoS-like behavior using traffic spikes, repeated requests, high connection counts, or CSF/Imunify360 blocking patterns. It should detect suspicious file or malware activity through Imunify360 or file scanning hooks where available. It should also support backup monitoring, alerting when backup jobs fail, are incomplete, or stop due to permission issues, disk space issues, or file access errors.

Design the system so that it uses a layered architecture. The first layer is the collector layer, which gathers raw data from logs, system commands, service status calls, and configuration sources. The second layer is the parser and normalizer layer, which transforms raw outputs into structured events. The third layer is the correlation and rules engine, which groups events by account, IP, domain, process, time window, and severity. The fourth layer is the incident engine, which decides whether an event is informational, suspicious, or critical and assigns a confidence score. The fifth layer is the notifier layer, which generates alerts, dashboard notifications, and optional email or webhook events. The sixth layer is the web dashboard layer, which shows real-time and historical data in a readable operator interface. The seventh layer is the persistence layer, which stores snapshots, incidents, events, service status, thresholds, and audit logs. Each layer must be replaceable without rewriting the whole system.

Use a robust backend stack suitable for a local server dashboard on port 8122. The backend must expose both a web UI and a JSON API. It must support authentication, role-based access, and a secure session or token system. The dashboard should include a clean summary page, live alerts page, service health page, resource usage page, logs page, database page, backups page, security page, and settings page. The summary page should show current load, top issues, active alerts, service status, and recent incidents. The logs page should allow filtering by service, severity, hostname, cPanel account, IP, domain, or time range. The alert detail page should show evidence, parsed snippets, correlated entities, and recommended response actions. The settings page should allow editing thresholds, enabled modules, log paths, scan intervals, and notification preferences. Every page should be designed to load quickly and remain usable on high-load servers.

The backend must support near-real-time monitoring but must not overwhelm the server. It should use configurable polling intervals, incremental log tailing, file offset tracking, and caching to avoid rereading entire logs. It should be efficient with memory and CPU. It should avoid expensive full-disk scans on every cycle. Where possible, it should read only new log lines and only perform deeper analysis when a trigger threshold is crossed. It should support a lightweight mode for low-resource systems and a full-analysis mode for detailed troubleshooting. It should gracefully handle very large logs, rotated logs, compressed logs where feasible, and missing or truncated files. It should deduplicate repeated alerts and suppress noisy repeats using a configurable cooldown and aggregation system.

The system must treat optional integrations carefully. If Imunify360 is not installed, security scanning modules should be disabled but the rest of the platform must continue functioning. If CSF is absent, firewall-related correlation should be disabled but the platform must still analyze logs. If Backuply is absent, backup monitoring should be hidden or marked unavailable. If WP Toolkit is absent, WordPress-specific checks should be skipped. If CloudLinux OS is not present, LVE-related signals should be marked unavailable but the monitoring agent should still analyze system and web logs. This optionality must be visible in both configuration and UI, and the system should never require all modules to exist in order to run.

Create a strong incident model. Each incident must include an ID, timestamp, severity, category, summary, detailed explanation, likely affected account or service, source evidence, raw event references, confidence score, suggested remediation, status, and resolution notes. Support incident states such as new, acknowledged, investigating, resolved, suppressed, and false positive. The dashboard must allow operators to acknowledge, comment on, suppress, or resolve incidents. It should also allow exporting incident data in JSON and CSV formats for reporting. Include an audit trail for all operator actions.

The log analysis engine must understand common hosting patterns. It should attempt to map IPs, domains, usernames, and account ownership using cPanel data, userdata mappings, domain lists, process lists, and service logs where available. It should recognize that an account can have multiple addon domains, subdomains, and parked domains. When possible, it should link activity to the correct cPanel user. For database abuse, it should map MySQL users and databases back to the related account. For web abuse, it should associate request floods or suspicious paths with the originating domain and user. For database or web errors, it should present short excerpts from the relevant logs and not dump huge unreadable blocks by default.

Add smart alerting rules for common hosting threats and problems. Alert on repeated 500 errors, 503 spikes, suspicious file upload or execution patterns, high open connections, abusive cron-like behavior, runaway PHP workers, excessive entry processes, abnormal RSS memory growth, repeated Imunify360 detections, repeated CSF blocks from the same IP, brute-force login attempts, slow query log spikes, database connection failures, backup job failures, disk space warnings, inode exhaustion, permission-related backup failures, and service restarts or crashes. Each rule should have a default threshold and an explanation, but users must be able to override thresholds and mute rules in config or the dashboard. Alert output should always be plain-language and actionable.

The web dashboard on port 8122 must be secure. It should support at least basic authentication or token-based login, configurable session timeouts, CSRF protection, rate limiting for sensitive routes, and optional IP allowlisting. It must bind only to the configured interface and must not expose unsafe administrative endpoints. The dashboard should never execute shell input from the browser unsafely. Any command execution must be preapproved, parameterized, and restricted to safe server-health actions only. Prefer read-only monitoring by default. If any active remediation actions are included, they must be opt-in, heavily guarded, logged, and separated from the monitoring path.

The agent should include a notification subsystem. Support local alerts on the dashboard, email notifications, optional webhook delivery, and optional integration hooks for future systems. Notifications should be grouped and rate-limited to avoid spam. A burst of repeated identical events should be summarized rather than repeatedly fired. Notifications must include severity, summary, impacted service, evidence, and recommended next steps. The system should also allow escalation policies based on severity and repeat count.

Design the storage layer so it is simple and reliable. Use a local database such as SQLite for single-server deployments, with the option to upgrade to MySQL/MariaDB if needed later. Persist configuration, service inventory, event history, parsed incidents, operator actions, alert acknowledgments, and scan checkpoints. Include migrations and schema versioning. Ensure the database is resilient to restarts and can recover cleanly if the agent crashes. Keep log offsets so tailing resumes correctly after reboot.

The deployment experience must be one command. Provide a single installer script that can run on a clean supported server or an existing cPanel server. The script should detect the OS family, install required packages, create a dedicated system user if appropriate, write the application files, generate the config file, set up the systemd unit, open or instruct about port 8122 if needed, start the service, and print the dashboard URL and login details or token creation instructions at the end. The installer should support unattended mode with environment variables or flags, as well as an interactive mode for first-time setup. The script should be idempotent and safe to rerun. If the system is already installed, it should upgrade or repair instead of duplicating files.

Build with maintainability in mind. Use clear project structure, clean separation of concerns, meaningful file names, type-safe code where practical, logging with levels, config validation, and modular plugins for each service integration. Every module should be independently testable. Include unit tests for parsers, correlation rules, and incident generation. Include a dry-run mode for validation and a debug mode for verbose logs. Include a healthcheck endpoint for systemd and external monitoring. Include version output and self-diagnostics. The project should be easy for a future engineer to extend with new modules such as email queue monitoring, malware scan integrations, or Nginx support.

Now implement the system as a full production-ready project. Deliver the complete source code, the bootstrap installer script, the backend server, the dashboard frontend, the API endpoints, the config template, the systemd service file, the database schema, the default rules engine, and the documentation needed to deploy and operate it. The result should be ready to run on port 8122 with one command using the installer script, and it must work even when some optional services are missing. Prefer safe defaults, clear logs, and readable output over cleverness.
